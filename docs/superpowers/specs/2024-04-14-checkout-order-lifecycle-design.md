# Checkout and Order Lifecycle Design

## 1. System Overview

This document outlines the design for integrating a comprehensive checkout flow and order lifecycle management into the Swipify platform. It covers the database schema, backend APIs, and the Flutter frontend UI flow, ensuring that all core requirements, including address management, shipping calculation, immutable order snapshots, and a robust order lifecycle, are met.

## 2. Core Requirements Addressed

*   **User selects address → shipping option → shipping fee calculated → included in order:** Handled by Flutter UI flow, backend shipping API, and order creation.
*   **NO hardcoded shipping fees:** Enforced by backend shipping calculation API.
*   **ALL calculations must be backend-controlled:** Ensured by moving shipping fee calculation logic to the backend.
*   **NO checkout without address and shipping option:** Enforced by frontend validation and backend API requiring these details.
*   **Orders must store a full shipping snapshot (immutable):** Implemented by storing `SelectedShippingOption` and `shipping_address` directly within the order document.
*   **System must be scalable for future logistics APIs:** The current shipping API design allows for integration of more complex logic or external APIs.
*   **Backend is the single source of truth:** All critical business logic and data manipulation are confined to the backend.
*   **No inconsistent shipping/order data:** Implemented through snapshotting.
*   **No missing validation:** Addressed with Pydantic models for API requests and service-level validation.
*   **No direct frontend manipulation of order status:** Order status transitions are strictly controlled by backend service.

## 3. Order Lifecycle Definition

*   **States:** `pending` → `paid` → `processing` → `shipped` → `in_transit` → `delivered` → `completed`
*   **Special States:** `cancelled` (allowed from `pending`, `paid`, `processing`)
*   **Rules:**
    *   No skipping states.
    *   Only backend/system can update status.
    *   Every transition must be logged.

### 3.1 Order State Ownership and Transitions

Order status transitions are exclusively managed by the backend system. This ensures data integrity and adherence to the defined lifecycle rules. Frontend applications or direct database manipulations by unauthorized entities are prevented from changing order statuses. All state changes must occur through the designated backend services and APIs, which will enforce valid transitions defined in `VALID_ORDER_TRANSITIONS`.

### 3.2 Separation of Payment Status and Order Status

The system maintains a clear separation between `order status` (e.g., `pending`, `processing`, `shipped`) and `payment_status` (e.g., `unpaid`, `paid`, `failed`, `refunded`).
*   **Order Status:** Reflects the logistical and fulfillment progress of the order.
*   **Payment Status:** Reflects the financial state of the payment for the order.
A change in `payment_status` (e.g., from `unpaid` to `paid`) typically triggers an `order status` transition (e.g., from `pending` to `paid`), but they are managed independently. A payment might fail while an order is still `pending`, or an order might be `cancelled` while the payment remains `paid` (requiring a separate refund process).

## 4. Database Schema (Firestore Structure)

### `users/{user_id}/addresses/{address_id}` (Subcollection)

*   **Purpose:** Stores multiple shipping/delivery addresses for each user.
*   **Document Fields for `address_id`:**
    *   `id`: `string` (Address ID, auto-generated by Firestore or provided by client if needed)
    *   `user_id`: `string` (ID of the user this address belongs to)
    *   `full_name`: `string` (Recipient's full name)
    *   `phone`: `string` (Recipient's phone number, validated by `^\+63\d{10}$`)
    *   `region`: `string`
    *   `city`: `string`
    *   `barangay`: `string`
    *   `street`: `string`
    *   `postal_code`: `string` (5-digit postal code)
    *   `is_default`: `boolean` (True if this is the user's default address)
    *   `created_at`: `timestamp`
    *   `updated_at`: `timestamp`

### `orders/{order_id}` (Top-level Collection)

*   **Purpose:** Stores details of each placed order.
*   **Document Fields for `order_id`:**
    *   `id`: `string` (Order ID)
    *   `user_id`: `string`
    *   `seller_id`: `string`
    *   `items`: `array<map>` (Snapshot of `OrderItem`s)
    *   `total_price`: `float` (Total price including product price and shipping fee)
    *   `status`: `string` (Current order status, `OrderStatus` enum value)
    *   `payment_status`: `string`
    *   `created_at`: `timestamp`
    *   `updated_at`: `timestamp`
    *   **`shipping_details`: `map` (Snapshot of selected shipping option)**
        *   `id`: `string`
        *   `name`: `string`
        *   `fee`: `float`
        *   `estimated_days_min`: `int`
        *   `estimated_days_max`: `int`
    *   **`shipping_address`: `map` (Snapshot of selected address at time of order)**
        *   `id`: `string` (Snapshot of address ID)
        *   `full_name`: `string`
        *   `phone`: `string`
        *   `region`: `string`
        *   `city`: `string`
        *   `barangay`: `string`
        *   `street`: `string`
        *   `postal_code`: `string`
        *   `is_default`: `boolean` (Snapshot of its default status at the time of order)

### `orders/{order_id}/status_history/{history_id}` (Subcollection)

*   **Purpose:** Immutable log of all status transitions for a given order.
*   **Document Fields for `history_id`:**
    *   `timestamp`: `timestamp`
    *   `old_status`: `string` (`OrderStatus` enum value)
    *   `new_status`: `string` (`OrderStatus` enum value)
    *   `updated_by`: `string` (e.g., "system", "user_id")
    *   `notes`: `string` (Optional, e.g., "Order created", "Manual override")

## 5. Backend APIs (FastAPI)

### Address Management APIs (`backend/app/routes/address.py`)

*   **`POST /address/` (Create Address):**
    *   **Request Body:** `AddressCreateRequest` (Pydantic model)
        *   `user_id`: `str`
        *   `full_name`: `str`
        *   `phone`: `str` (Validated by `^\+63\d{10}$`)
        *   `region`: `str`
        *   `city`: `str`
        *   `barangay`: `str`
        *   `street`: `str`
        *   `postal_code`: `str` (Validated by `^\d{5}$`)
        *   `is_default`: `bool` (Optional, defaults to `false`. If `true`, existing default for user will be unset.)
    *   **Response:** `AddressResponse`
    *   **Logic:** Create, validate, handle default status transactionally.

*   **`GET /address/user/{user_id}` (Get User Addresses):**
    *   **Path Parameter:** `user_id`: `str`
    *   **Response:** `List[AddressResponse]`
    *   **Logic:** Retrieve addresses from `users/{user_id}/addresses`.

*   **`PUT /address/{user_id}/{address_id}` (Update Address):**
    *   **Path Parameters:** `user_id`: `str`, `address_id`: `str`
    *   **Request Body:** `AddressUpdateRequest` (Pydantic model, all fields optional for partial update)
        *   `full_name`: `str`
        *   `phone`: `str`
        *   `region`: `str`
        *   `city`: `str`
        *   `barangay`: `str`
        *   `street`: `str`
        *   `postal_code`: `str`
        *   `is_default`: `bool` (If `true`, unsets other defaults.)
    *   **Response:** `AddressResponse`
    *   **Logic:** Update, validate, handle default status transactionally.

*   **`DELETE /address/{user_id}/{address_id}` (Delete Address):**
    *   **Path Parameters:** `user_id`: `str`, `address_id`: `str`
    *   **Response:** `HTTP 204 No Content`
    *   **Logic:** Delete address document.

*   **`PUT /address/{user_id}/{address_id}/set-default` (Set Default Address):**
    *   **Path Parameters:** `user_id`: `str`, `address_id`: `str`
    *   **Response:** `AddressResponse`
    *   **Logic:** Set specified address as default, unset previous default transactionally.

### Checkout Integration (`backend/app/routes/orders.py` and `backend/app/services/order_service.py`)

*   **`POST /orders/` (Create Order from Cart Checkout):**
    *   **Request Body:** `OrderCreateRequest` (existing, with new `selected_address_id`)
        *   `selected_address_id`: `str` (ID of the selected address from `users/{user_id}/addresses`)
    *   **Logic:** Fetch full `AddressModel` using `selected_address_id`, store snapshot as `shipping_address` in order.

*   **`POST /orders/buy-now` (Create Buy Now Order):**
    *   **Request Body:** `BuyNowRequest` (existing, with new `selected_address_id`)
        *   `selected_address_id`: `str` (ID of the selected address from `users/{user_id}/addresses`)
    *   **Logic:** Similar to `create_order_service`, fetch `AddressModel` and store its snapshot as `shipping_address`.

### Shipping Calculation API (`backend/app/routes/shipping.py` - existing)

*   No changes to this API, as it's already implemented.

### 5.1 Payment Webhook Flow

Payment processing will typically involve a third-party payment gateway. After a user initiates payment (e.g., clicks "Place Order"), the frontend will interact with the payment gateway. Upon successful payment, the gateway will send a webhook notification to a dedicated backend endpoint (e.g., `POST /webhooks/payment-success`). This webhook endpoint will:
1.  Verify the webhook's authenticity.
2.  Retrieve the associated order (using a payment ID or order ID from the webhook payload).
3.  Update the order's `payment_status` to "paid" and trigger a status transition from `pending` to `paid` using `update_order_status_service`.
4.  Handle any potential failures or retries.

### 5.2 Checkout Idempotency

To prevent duplicate orders from multiple submission attempts (e.g., user double-clicks "Place Order"), the checkout process needs to be idempotent. This can be achieved by:
1.  **Frontend:** Disabling the "Place Order" button immediately after the first click and showing a loading indicator.
2.  **Backend:** Implementing an idempotency key mechanism. The frontend would generate a unique idempotency key (e.g., a UUID) for each checkout attempt and send it in the `POST /orders/` request header (e.g., `X-Idempotency-Key`). The backend would store this key and, for subsequent requests with the same key, return the result of the original processing without re-executing the order creation logic.

### 5.3 Locking Rules for Shipping/Address Snapshots

Once an order is successfully created in the `orders` collection, the `shipping_address` and `shipping_details` fields are considered immutable snapshots of the data at the time of order placement.
*   These fields **SHOULD NOT** be modified after the order is created.
*   Any changes to a user's address book or shipping option configurations in the system will **NOT** affect existing orders. This ensures that the historical record of an order remains accurate, reflecting the exact conditions under which it was placed.

### 5.4 Firestore Transaction Usage

Critical operations that involve multiple related database writes or require strong consistency will utilize Firestore transactions. This includes:
*   **Creating a new address with `is_default=true`:** Unsetting a previous default address and setting the new one as default must happen atomically to ensure only one default address exists per user.
*   **Setting an address as default via `set-default` API:** Similar to creation, this requires atomic updates to multiple documents.
*   **Order Creation:** The creation of the order document and its initial status log to `status_history` subcollection should ideally be part of a single transaction, though Firestore's eventual consistency for subcollections might allow for separate writes if a strict single transaction is too complex with the current Firestore client. For `create_order_service` and `buy_now_service`, ensuring the order and its initial status log are consistent is paramount.

### 5.5 Multi-Seller Order Handling

The current design assumes a single order is created per seller. If a user's cart contains items from multiple sellers, the frontend is expected to group these items by seller and initiate a separate order creation request for each seller. Each such order will have its own lifecycle, shipping details, and payment status, managed independently. This simplifies the order fulfillment process for individual sellers and avoids complex split-shipment logic within a single order document.

### 5.6 Checkout Session Boundaries

A "checkout session" is implicitly managed by the frontend's navigation through the multi-step checkout flow and the `CheckoutProvider`'s state.
*   **Cart Persistence:** The user's cart items are assumed to be persisted (e.g., in a `CartProvider` or locally) before entering the checkout flow.
*   **Session Start:** A checkout session effectively begins when the user enters the `CheckoutScreen`.
*   **Session Data:** The `CheckoutProvider` holds the transient state of the current checkout session, including `selectedAddress`, `selectedShippingOption`, and `cartItems`.
*   **Session Completion:** A session concludes upon successful order placement, at which point the cart should be cleared (as handled in `create_order_service`), and the user is typically navigated to an order confirmation screen.
*   **Session Interruption:** If the user abandons the checkout flow, the `CheckoutProvider`'s state is lost, but the underlying cart remains persistent for a future session.

## 6. Frontend (Flutter) UI Flow

### 1. Address Management Screen (New Screen)

*   **Purpose:** Allow users to view, add, edit, and delete their addresses, and set a default address.
*   **Location:** `frontend/lib/screens/address_management_screen.dart`
*   **Components:** `AddressCard` widgets for display, "Add New" button, edit/delete/set-default actions within cards.
*   **State Management:** `AddressProvider` to manage address list.
*   **Integration:** Calls new backend Address Management APIs.

### 2. Checkout Multi-Step Flow (`frontend/lib/screens/checkout_screen.dart` - enhancements)

*   **Navigation:** Multi-step navigation (e.g., `PageView`).
*   **Step 1: Select Address:**
    *   Uses `AddressSelectionWidget`.
    *   `CheckoutProvider` updates `selectedAddress`.
*   **Step 2: Select Shipping Option:**
    *   Uses `ShippingOptionsSelectionWidget`.
    *   Fetches options based on `selectedAddress` and cart items.
    *   `CheckoutProvider` updates `selectedShippingOption`.
*   **Step 3: Review Order:**
    *   Uses `OrderSummaryWidget`.
    *   Displays full order summary, including selected address and shipping details.
    *   "Place Order" calls `checkoutProvider.placeOrder()`.

## 7. Seller Order View Logic

*   The existing `get_seller_orders` endpoint in the backend, which returns `OrderResponse`, is sufficient. `OrderResponse` will now include the `shipping_details` and `shipping_address` snapshots, which can be displayed in the seller's UI. No new backend API or dedicated frontend screen is needed at this time.

## 8. Implementation Steps (High-Level)

1.  **Backend: Create Address Management Models** (`AddressCreateRequest`, `AddressUpdateRequest`, `AddressResponse` in `backend/app/models/address.py`).
2.  **Backend: Implement Address Service** (`backend/app/services/address_service.py`) to handle CRUD and default logic, interacting with Firestore, ensuring transactional integrity for default address management.
3.  **Backend: Implement Address Routes** (`backend/app/routes/address.py`) using the service.
4.  **Backend: Update Order Models** (`OrderCreateRequest`, `BuyNowRequest`, `OrderResponse` in `backend/app/models/order.py`) to include `selected_address_id` and `shipping_address` snapshot.
5.  **Backend: Update Order Service** (`create_order_service`, `buy_now_service` in `backend/app/services/order_service.py`) to fetch `shipping_address` and store its snapshot, ideally within a Firestore transaction for order creation.
6.  **Frontend: Create Address Models** (`address_model.dart` is existing, but might need minor adjustments).
7.  **Frontend: Create Address Service** (`address_service.dart`) to call backend Address Management APIs.
8.  **Frontend: Create Address Management Screen** (`address_management_screen.dart`) with associated widgets.
9.  **Frontend: Enhance CheckoutProvider** to handle `selected_address_id` for order creation.
10. **Frontend: Update CheckoutScreen** to integrate `AddressManagementScreen` (e.g., as part of the flow for selecting an address).
